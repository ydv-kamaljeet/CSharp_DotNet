1) What is a branch?

A branch is like a separate “timeline” of your project. You create a branch to work on a feature or bugfix without breaking the main code.

Visual idea (simplified)

main:     A --- B --- C
                 \
feature/login:    D --- E

Meaning: commits D and E are added in the feature branch. main is still safe at C.

In GitHub projects: most teams keep main protected and do changes via feature branches + Pull Requests (PR).

2) Setup (one-time)

Run these once on your machine (recommended):

git config --global user.name "Your Name"
git config --global user.email "you@example.com"
git config --global init.defaultBranch main
If you are using GitHub, enable two-factor auth and prefer SSH or Personal Access Token for pushing.

3) Create & switch branches

GoalCommand
See branchesgit branch
Create a branchgit branch feature/login
Switch branchgit switch feature/login
Create + switch (one step)git switch -c feature/login
Old way (still valid)git checkout -b feature/login

Mini demo

# Start from main
git switch main
git pull

# Create new branch
git switch -c feature/login

# Work + commit
git add .
git commit -m "Add login UI"

Common mistake: Creating a branch from an outdated main. Always git switch main + git pull first.

4) Merge (fast-forward vs merge commit)

A) Fast-forward merge (no divergence)

main:    A --- B
               \
feature:         C --- D

If main didn’t move after branching, Git can “fast-forward” main to the feature tip.

git switch main
git merge feature/login

B) Merge commit (branches diverged)

main:    A --- B --- X
               \       \
feature:         C --- D  (merge commit M)

If both branches have new commits, Git creates a merge commit.

git switch main
git merge feature/login
Team tip: Many teams use GitHub PRs with “Squash and merge” (one clean commit) or “Rebase and merge”.

5) Merge conflicts

A conflict happens when Git cannot decide which change to keep (same lines edited in both branches).

How to resolve conflicts (safe steps)

  1. Run merge/rebase. Git will stop and mark conflicts.
  2. Open conflict files and choose the correct lines.
  3. Mark resolved: git add <file>
  4. Finish:
    • For merge: git commit
    • For rebase: git rebase --continue
# Example: after merge conflict
git status
# edit files (remove conflict markers)
git add .
git commit
If you feel stuck, you can cancel:
  • Cancel merge: git merge --abort
  • Cancel rebase: git rebase --abort

6) Rebase (rewrite history)

Rebase “replays” your commits on top of another branch to create a clean linear history.

Before and after

Before:
main:     A --- B --- C
                 \
feature:           D --- E

Rebase feature onto main:
main:     A --- B --- C
                       \
feature:                D' --- E'

Important: Rebase creates new commits (D', E') with new hashes.

Common use: keep your feature branch updated

git switch feature/login
git fetch origin
git rebase origin/main
Golden rule: Do not rebase commits that were already pushed and used by others (shared history), unless your team explicitly allows it and you know the impact.

Interactive rebase (clean your commits)

# Reorder, squash, or edit last 5 commits
git rebase -i HEAD~5

Interactive rebase is used to squash messy “fix typo” commits into one neat commit before PR.

7) Cherry-pick (copy commits)

Cherry-pick copies a specific commit from one branch and applies it onto another branch.

Real-world use cases

  • Urgent hotfix in main needs to go into release branch
  • You accidentally committed to the wrong branch and want to move a commit
  • Only one commit from a large feature is needed elsewhere

Example: pick one commit into main

# Find commit hash from another branch
git log --oneline feature/login

# Switch to target branch
git switch main

# Copy that commit here
git cherry-pick <commit-hash>

Cherry-pick a range of commits

# Pick commits from A (exclusive) to B (inclusive)
git cherry-pick <A>..<B>
Conflicts can happen during cherry-pick too. Resolve + git add . then:
  • Continue: git cherry-pick --continue
  • Abort: git cherry-pick --abort
Cherry-pick vs Merge:
  • Merge brings the whole branch history
  • Cherry-pick brings only selected commits

8) Revert vs Reset

CommandWhat it doesSafe for pushed commits?
git revertMakes a new commit that undoes a previous commitYes ✅
git resetMoves branch pointer backwards (can delete commits locally)No ⚠️ (rewrites history)

A) Revert (recommended in teams)

# Undo a specific commit safely
git revert <commit-hash>

B) Reset (local cleanup)

# Keep changes staged (rare)
git reset --soft HEAD~1

# Keep changes unstaged (common)
git reset --mixed HEAD~1

# Delete commit + changes (danger!)
git reset --hard HEAD~1
Never run git reset --hard unless you are 100% sure you don't need those changes.

9) Stash (save work temporarily)

Stash is like putting your unfinished work into a drawer, so you can switch branches quickly.

# Save current changes
git stash push -m "WIP: login form"

# See stashes
git stash list

# Apply latest stash (keep stash)
git stash apply

# Apply and remove stash
git stash pop

# Drop a stash
git stash drop stash@{0}
If you often stash, use meaningful messages (-m) so you can find them easily.

10) Rename & delete branches

# Rename current branch
git branch -m feature/login feature/auth-login

# Delete local branch (safe: refuses if unmerged)
git branch -d feature/auth-login

# Force delete (danger)
git branch -D feature/auth-login
Safe delete (-d) is preferred. Force delete (-D) only if you're sure.

11) Remote branches + GitHub

When you push a branch to GitHub, others can see and collaborate.

# Push current branch to origin and set upstream
git push -u origin feature/login

# List remote branches
git branch -r

# Fetch updates from remote
git fetch origin

# Pull updates into current branch
git pull

Typical GitHub flow

1) Create branch: feature/login
2) Push to GitHub
3) Open Pull Request
4) Code review + checks
5) Merge PR into main
6) Delete remote branch

12) Pull Requests workflow (industry standard)

Most teams follow this process:

  1. Update main: git switch main + git pull
  2. Create feature branch: git switch -c feature/x
  3. Commit small logical changes
  4. Push: git push -u origin feature/x
  5. Create PR in GitHub
  6. Fix review comments (push more commits)
  7. Merge via Squash/Rebase/Merge (as per team policy)
Squash and merge: makes PR history clean (one commit in main) — great for beginner teams.

13) Best practices

  • Branch naming: feature/, bugfix/, hotfix/, release/
  • Small commits: one change per commit with clear message
  • Sync often: rebase or merge from main regularly
  • Protect main: require PR review + CI checks in GitHub settings
  • Avoid force push unless you understand the impact
Rule of thumb: If the branch is shared → prefer merge or revert. If local only → you can rebase and reset safely.

14) Cheatsheet

# Branch basics
git branch
git switch -c feature/x
git add .
git commit -m "message"
git push -u origin feature/x
# Merge (on main)
git switch main
git pull
git merge feature/x

# Rebase (on feature branch)
git switch feature/x
git fetch origin
git rebase origin/main
# Cherry-pick one commit
git switch main
git cherry-pick <hash>

# Revert (safe)
git revert <hash>

# Reset (local rewrite)
git reset --mixed HEAD~1
git reset --hard  HEAD~1
Tip: Use git log --oneline --graph --decorate --all to visualize your branches.

15) Exercises (Beginner → Expert)

Beginner

  1. Create a repo, make 2 commits on main.
  2. Create feature/readme, add 2 commits, merge into main.
  3. Create bugfix/typo, fix a typo, open a PR (even for practice).

Intermediate

  1. Create two branches that modify the same file lines, trigger a conflict, resolve it.
  2. Do an interactive rebase to squash 3 commits into 1.
  3. Use stash to switch branches without committing.

Expert (real-world)

  1. Simulate a hotfix: create hotfix/payments, commit fix, merge to main.
  2. Create release/v1, then cherry-pick the hotfix commit into release/v1.
  3. Undo the hotfix on release/v1 using git revert (not reset).
